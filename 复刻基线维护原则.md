# 复刻基线维护原则

> reBot-DevArm 复刻工程的依赖管理与同步规则
>
> 本文档是项目同步、升级、维护决策的**最高依据**，优先级高于任何 AI 记忆和临时讨论结果。

---

## 一、项目定位

**本仓库是复刻集成工程，不是单一软件包维护。**

**核心目标**：持续跟踪上游最新进展，同时维护一套可回滚、可复现、已验证的复刻基线。

**一句话原则**：追最新，但不盲目追最新；更新必须记录来源、原因、验证结果和回滚点。

### 复刻基线的三层结构

```
sync/<repo>-<日期> 分支  ←  追最新、试同步、试升级（可随时丢弃）
       ↓ 验证通过
       ↓
main 分支               ←  已经验证过的"最新集成状态"（滚动更新）
       ↓ 阶段性固化
       ↓
baseline-<日期> tag     ←  阶段性可复现基线（永不变动）
```

| 衡量维度 | 软件包维护 | 复刻集成工程（本项目） |
|---------|----------|--------------------|
| 上游更新 | 跟随升级 | 必须检查 + 评估 + 验证 |
| main 分支 | 跟随上游 | 只接受验证过的最新 |
| 升级动机 | "有更新就升" | "有更新先评估，验证后再升" |
| 关键产出 | 跑得快、功能全 | 每个版本停在哪、为什么停、验证记录 |
| 失败处理 | 修 bug 升级 | 回滚到 baseline tag |
| 节奏控制 | 被动跟随 | 主动追新 + 主动门控 |

## 二、核心规则

### 规则 1：所有核心依赖必须 fork 到 fanhao375

**原因**：fork 不是为了改源码，而是为了建立锁定权和可访问性。

| 风险 | 不 fork 的后果 |
|------|--------------|
| 锁定权 | 上游强制 push 后你的 commit 可能消失 |
| 可访问性 | 上游被删/转私有，submodule 全部失效 |
| 审计权 | 不知道每个 commit 是谁、什么时候改的 |
| 节奏控制 | 被上游"必须升级"的节奏裹挟 |

### 规则 2：主仓库 submodule 永远只指向自己的 fork

**.gitmodules 单源原则**：
- ✅ origin = fanhao375/<repo>（submodule 实际跟踪目标）
- ✅ upstream = 原作者仓库（只在 fork 内部维护）
- ❌ 主仓库 .gitmodules 不出现原作者地址

理由：避免"哪天升级到底从哪个 remote 拉的"这种混乱。

### 规则 3：同步前必须先检查

**触发检查的时机**（不是每次进项目）：
- 准备做同步操作前
- 准备做正式 commit 前
- 阶段总结前
- 用户明确要求

**检查内容**：
```bash
# 主仓库
git fetch upstream && git log HEAD..upstream/main --oneline

# 各 submodule
git submodule foreach 'git fetch && git log HEAD..@{upstream} --oneline'
```

### 规则 4：不同步也要记录

每次检查上游，无论是否同步，都必须在 `上游更新记录/<日期>_同步详情.md` 中记录：

**必填字段**：
- 检查日期
- 仓库
- 当前本地 commit
- 上游最新 commit
- 新增提交数
- 主要变化
- 是否同步：是 / 否
- **如不同步**：不同步原因 + 下次触发条件
- **如同步**：验证结果

理由：6 个月后回来看，不用重新调研一遍"当时为啥停这"。

### 规则 5：升级走 sync 分支

submodule 升级**绝不直接在 main 上做**，必须走流程：

```
1. fork 内合入 upstream/main（在 fanhao375/<repo> 内操作）
2. 主仓库新建 sync/<repo>-<日期> 分支
3. 在 sync 分支更新 submodule 指针
4. 跑最小验证（每个仓库门槛见下文）
5. 通过后合入 main
6. 在主仓库打 baseline tag（baseline-<日期>）
```

**理由**：sync 分支可以随时丢弃，main 只接受验证过的最新；baseline tag 永不变动，永远可回滚。

### 规则 6：commit 分层

不同性质的工作必须分开 commit，不能混合：

| 工作类型 | commit 前缀示例 |
|---------|---------------|
| 策略变更 | `确立 XX 原则` / `调整 XX 策略` |
| 主仓库同步 | `同步 Seeed 主仓库 <日期>` |
| Submodule 升级 | `升级 MotorBridge 到 v0.2.4` |
| 学习笔记 | `补充 XX 学习笔记` |
| 操作日志 | `更新操作日志` |
| 文档整理 | `整理 XX 文档` |

理由：以后 bisect 或回滚时能精准定位。

### 规则 7：永不擅自 push

**任何对 GitHub 远程仓库的写操作**（push / 改 .gitmodules URL / fork 操作）都需要用户明确确认。

## 四、文档优先级

**主依据顺序（从高到低）**：

1. **`复刻基线维护原则.md`**（本文件）— 最高优先级，所有规则的源头
2. **`项目总览.md`** — 项目结构、AI 接手提示
3. **`操作日志.md`** — 历史操作时序
4. **`上游更新记录/`** — 每次同步检查的具体记录
5. **`学习笔记.md`** — 学习过程、踩坑、原理
6. **AI 记忆**（`~/.claude/projects/.../memory/`）— **仅作辅助参考**

**为什么 AI 记忆排在最后**：
- AI 记忆有过期标记（5 天就提示"may be outdated"）
- 不同 AI 之间记忆不互通
- 用户和协作者看不到 AI 记忆
- 仓库文档才是真正的项目交付物

**记忆系统的合理用途**：
- 辅助 AI 快速定位仓库文档
- 记录 AI 个人协作偏好（不在文档里写的）
- **不能作为业务决策的主依据**

## 五、AI 协作规则

### 必须做

| 时机 | 动作 |
|------|------|
| 做同步操作前 | 检查主仓库和 submodule 状态 |
| 做正式 commit 前 | 检查工作区状态、文档同步性 |
| 检查到上游更新 | 先记录到 `上游更新记录/`，再决策 |
| 决定升级 submodule | 走 sync 分支流程，不在 main 上直接做 |
| 不同步上游 | 写明不同步原因 + 下次触发条件 |
| 完成阶段性工作 | 提醒用户更新对应文档 |

### 必须不做

| 红线 | 后果 |
|------|------|
| ❌ 擅自 push 到任何远程 | 远程状态不可控 |
| ❌ 擅自升级 submodule（改 gitlink） | 可能搞坏复刻链路 |
| ❌ 擅自改 .gitmodules URL | 影响 fork 关系，必须用户确认 |
| ❌ 只信 AI 记忆做决策 | 记忆可能过期，必须查仓库文档 |
| ❌ 把多类工作混在一个 commit | 影响后续 bisect 和回滚 |

### 用户授权后可以做

| 操作 | 触发条件 |
|------|---------|
| ✅ 改 .gitmodules URL | 用户确认 fork 完成 |
| ✅ 升级 submodule 指针 | 用户确认走 sync 流程 |
| ✅ push 到远程 | 用户明确说"push 一下" |

## 七、长期治理目标

最终愿景（fork 完成后的状态）：

```
fanhao375/reBot-DevArm（主仓库 fork）
├── upstream: Seeed-Projects/reBot-DevArm
├── software/MotorBridge
│   ├── origin: fanhao375/motorbridge
│   └── upstream: motorbridge/motorbridge
├── software/reBotArm_control_py
│   ├── origin: fanhao375/reBotArm_control_py
│   └── upstream: vectorBH6/reBotArm_control_py
└── 遥操作/StarArm_102
    ├── origin: fanhao375/Star-Arm-102
    └── upstream: servodevelop/Star-Arm-102
```

**复刻基线特征**：
- 所有依赖都在自己账号下，永不丢失
- 每个 submodule 锁定到验证过的 commit
- main 分支始终是"已验证的最新集成状态"
- baseline tag 标记阶段性可复现快照
- 每次升级都有记录、有理由、有验证、有回滚点
- 任何时候 clone 都能复现完整系统
