## 2026-04-29

### 达妙电机的 4 种控制模式（重要）

DM-J4310-2EC V1.2 支持 4 种工作模式，**同一时间只能用一个**：

| 编码 | 模式 | 帧 ID 偏移 | 控制内容 |
|------|------|-----------|---------|
| 1 | MIT 模式 | CAN_ID | 位置+速度+力矩混合 |
| 2 | 位置速度模式 | CAN_ID + 0x100 | 三环串级控制 |
| 3 | 速度模式 | CAN_ID + 0x200 | 双环控制 |
| 4 | 力位混控模式 (PVT) | CAN_ID + 0x300 | 位置+力矩限制 |

**模式切换的两种方式**：
1. **串口"写参数"**（永久保存）：电机自动重启，模式存到 flash，掉电不丢失
2. **CAN 命令写寄存器 0x0A**（临时切换）：电机不重启，但掉电丢失

注意：模式切换会清零所有指令值（位置、速度、KP、KD 等），从其他模式切换到位置控制前，建议先读取当前位置（寄存器 0x50），并在零速时切换。

### MIT 模式的来源和含义

**MIT 模式**这个名字来源于**麻省理工学院**的 **MIT Cheetah 四足机器人**项目（2018 年左右开源），后来被全球电机厂商沿用。

**核心公式**：
```
T = Kp × (p_des - p_actual) + Kd × (v_des - v_actual) + t_ff
```

一个公式同时干 3 件事：位置控制 + 阻尼控制 + 前馈力矩。通过调参数可以衍生出多种控制方式：

| 想要的效果 | Kp | Kd | t_ff |
|-----------|-----|-----|------|
| 纯位置控制 | ≠0 | ≠0 | 0 |
| 纯速度控制 | 0 | ≠0 | 0 |
| 纯力矩控制 | 0 | 0 | ≠0 |
| 柔顺位置控制 | 小 | 小 | 0 |
| 重力补偿位置控制 | ≠0 | ≠0 | 重力力矩 |

**警告**：位置控制时 **Kd 不能为 0**，否则电机会震荡甚至失控。

**为什么机器人都用 MIT 模式**：柔顺性好（碰到障碍会"让一下"）、稳定性好（Kd 防震荡）、统一接口、响应快。

### PVT 模式 = 力位混控模式

达妙的 PVT 是 **P**osition + **V**elocity + **T**orque（不是工业控制里的 P-V-Time）：

| 字节 | 内容 | 类型 | 范围 |
|------|------|------|------|
| D[0-3] | P：位置 | float | rad |
| D[4-5] | V：限速值（×100） | uint16 | 0-10000（对应 0-100 rad/s） |
| D[6-7] | T：电流标幺值（×10000） | uint16 | 0-10000（对应 0-1.0） |

**应用场景**：抓取易碎物品、柔性装配、人机协作。比"位置速度模式"多了**力矩限制**，更安全。

### CAN 反馈帧的解析方法

收到的 8 字节反馈帧格式：

| 字节 | 含义 |
|------|------|
| D[0] | `ID|ERR<<4` 低 4 位是 ID，高 4 位是错误码 |
| D[1-2] | POS 位置（16 位有符号定点数）|
| D[3-4] | VEL 速度（12 位）|
| D[4-5] | T 扭矩（12 位）|
| D[6] | T_MOS 驱动板温度（℃）|
| D[7] | T_Rotor 电机温度（℃）|

**例**：反馈 `36 D1 4D 7F F7 FB 24 22`
- D[0] = 0x36 = `0011 0110` → 高 4 位 = 3（**错误码 = 3，输出轴校准异常**），低 4 位 = 6（ID）
- D[6] = 0x24 = 36℃（驱动温度）
- D[7] = 0x22 = 34℃（电机温度）

**错误码对照表**：

| 状态码 | 含义 |
|--------|------|
| 0 | 失能 |
| 1 | 使能（正常）|
| 3 | 输出轴校准异常 |
| 4 | 传感器输出异常 |
| 5 | 电机编码器校准异常 |
| 8 | 超压 |
| 9 | 欠压 |
| A | 过电流 |
| B | MOS 过温 |
| C | 电机线圈过温 |
| D | 通讯丢失 |
| E | 过载 |

### CAN 命令实测：让电机以速度模式转动

测试电机：DM-J4310-2EC，CAN ID = 1

**步骤**：
1. **使能**：帧 ID `0x201`，数据 `FF FF FF FF FF FF FF FC`
2. **速度命令（5 rad/s）**：帧 ID `0x201`，数据 `00 00 A0 40 00 00 00 00`
3. **必须勾选"定时发送"**（间隔 10 ms），否则电机因 CAN Timeout 自动失能
4. **停止**：数据 `00 00 00 00 00 00 00 00`
5. **失能**：数据 `FF FF FF FF FF FF FF FD`

**速度对应的 IEEE 754 浮点字节（低位在前）**：

| 速度 (rad/s) | 十六进制 |
|-------------|---------|
| 1.0 | `00 00 80 3F` |
| 5.0 | `00 00 A0 40` |
| 10.0 | `00 00 20 41` |
| 20.0（接近最大）| `00 00 A0 41` |
| -5.0（反向）| `00 00 A0 C0` |

**最大速度**：DM-J4310 24V 版本空载约 200 rpm = 20.94 rad/s。

### 踩坑：达妙电机调试工具版本不对，导致串口乱码、ID 错乱

**问题现象**：
1. 串口正常显示开机信息，但执行校准等操作时出现乱码
2. CAN 反馈帧的电机 ID 一会儿是 6，一会儿是 4，与设置的 0x01 不符
3. 读取寄存器参数（如控制模式 0x0A）发送 `00 00 33 0A 00 00 00 00`，但收到的是普通反馈帧而非参数返回
4. 校准时调试助手提示"数据传输失败"
5. 反馈帧 D[0] = 0x36，解析出 ERR=3（输出轴校准异常）

**排查过程**：
- 一开始以为是硬件问题（串口线、电源、屏蔽）
- 也怀疑是电机损坏或 ID 配置问题
- 改了波特率、降了发送频率都不行
- 最后发现是**工具版本问题**

**根本原因**：
使用了错误的调试工具版本 `Debugging_Tools_v.1.6.8.8.exe`，实际应该用 `DMTool v2.1.6.0-FD加速(1).exe`

**工具对比**：

| 工具 | 版本 | 是否可用 |
|------|------|---------|
| Debugging_Tools_v.1.6.8.8.exe | V1.6.8.8 | ❌ 乱码、通信不稳定 |
| DMTool v2.1.6.0-FD加速(1).exe | V2.1.6.0-FD | ✅ 正常 |

**关键结论**：
- 达妙电机说明书提到"V2.0.0.0 及以上版本"，但实际 V1.x 系列工具完全不兼容新固件
- 必须用 **V2.x 系列**的 DMTool，且最好用带 **FD加速**的版本
- 工具位置：`hardware/测试工具/DMTool v2.1.6.0-FD加速(1).exe`

**怎么避免再踩坑**：
- 拿到达妙电机后，先确认工具版本 ≥ V2.0
- 出现以下症状立刻怀疑工具版本：
  - 串口乱码（特别是只在调试操作时乱码）
  - CAN 反馈 ID 与配置不一致
  - 读取参数无响应
  - 校准提示"数据传输失败"
- 不要花时间排查硬件，先换工具试

### Git 进阶操作（今天踩坑学到的）

#### 1. 中文 commit message 必须用文件，不能用 HEREDOC

**踩坑**：用 `git commit -m "$(cat <<'EOF' ... EOF)"` 提交中文 message 时，Windows shell 会出现字符截断乱码（"快照" → "快�"，"永不变动" → "永不变��"）。

**正确做法**：
```bash
# 1. 把 message 写到临时文件
# （可以用 Write 工具写或者 cat > 重定向）
# 2. grep "�" 验证文件无乱码
# 3. git commit -F .git/COMMIT_MSG_TEMP.txt
# 4. rm 删除临时文件
# 5. git log -1 --format=%B 验证 commit message 正常
```

**Why:** Windows 下 shell 处理长中文 HEREDOC 会有 buffer 截断问题，写文件再读取最稳。

#### 2. 撤销刚 commit 的：reset --soft 而不是 --amend

发现 commit message 写错了（乱码）时：
- ❌ 不要用 `git commit --amend`（项目规则禁止 amend，只创建新 commit）
- ✅ 用 `git reset --soft HEAD~1` 撤回，保留所有改动，重新 commit

**Why:** amend 会改写历史，对已 push 的 commit 危险。reset --soft + 重 commit 习惯更稳。

#### 3. submodule 切换到自己 fork 的完整流程

今天把 MotorBridge 和 StarArm_102 从原作者直连切到 fanhao375 fork，完整流程：

```bash
# (1) 改 .gitmodules（手动编辑或用 git config）
git config -f .gitmodules submodule.<path>.url <新URL>

# (2) 同步到 .git/config
git submodule sync

# (3) 进 submodule 内改 origin + 加 upstream
cd <submodule>
git remote set-url origin <fork URL>
git remote add upstream <原作者 URL>

# (4) 验证连通性
git fetch origin && git fetch upstream

# (5) 回主仓库 commit .gitmodules
cd ..
git add .gitmodules && git commit
```

**关键认知**：
- submodule 锁定的 commit 不变（fork 是从同一 commit 创建的）
- `.git/config` 里的 submodule URL 变化是本地的，不入 git
- `.gitmodules` 才是入库的唯一权威

#### 4. baseline tag 是复刻工程的"快照"

第一次理解 git tag 在复刻工程里的作用——不是发版本号，是**锁定一个已验证的状态**。

```bash
git tag -a baseline-2026-04-29 -m "复刻基线 2026-04-29
四个核心仓库的 submodule 锁定状态：
- ...
"
```

将来不管 main 怎么演进，`git checkout baseline-2026-04-29` 都能回到这个已知好的状态。

**Why:** 复刻工程不是"追最新"，是"维护一套可回滚的版本基线"。tag 是这个基线的物化体现。

#### 5. commit 必须分层

今天把 9 个 commit 分得很清楚：
- 整理电机文档
- 确立维护原则
- 修正核心定位
- 更新上游地址
- 切换 submodule
- 补充 Git 指南
- ...

**反面教材**：把"整理文档 + 修策略 + 升级 submodule + 改 .gitignore"塞一个 commit 里。

**Why:** 以后用 `git bisect` 定位 bug、用 `git revert` 回滚某个变更时，分层 commit 能精准定位到一类工作。

#### 6. 项目同步策略：检查文档是主依据，AI 记忆是辅助

今天讨论同步策略时一个关键修正：**仓库文档（`复刻基线维护原则.md`、`项目总览.md`）是项目最高依据**，AI 记忆只能做辅助指针。

**Why:**
- AI 记忆有过期标记（5 天就提示 may be outdated）
- 不同 AI 之间记忆不互通
- 用户和协作者看不到 AI 记忆
- 仓库文档才是真正的项目交付物

如果 AI 记忆和仓库文档冲突，永远以文档为准。

## 2026-04-12

### 项目认识

- reBot-DevArm 是 SeeedStudio 的开源 6DOF 机械臂项目，目标是降低具身智能学习门槛
- 有两个版本：B601 DM（达妙电机）和 B601 RS（RobStride 电机），外观结构一样，只是电机不同
- 截至今天，仓库里硬件资料齐全，软件代码刚发布（Python SDK + Pinocchio）

### 电机相关

- 两款电机都是**驱动一体化**（电机+FOC驱动器+双编码器集成在一个壳里），不需要单独买驱动器
- DM4340P（大电机）：27Nm 峰值扭矩，减速比 40:1，用在底座/肩部/大臂
- DM4310（小电机）：7Nm 峰值扭矩，减速比 10:1，用在小臂/腕部
- 4340P 比 4310 多一个"力位混控模式"，可以同时控制位置和限制力矩
- 所有电机通过 CAN 总线 @ 1Mbps 菊花链串联，协议是基于 MIT Mini Cheetah 的私有协议，不是 CANopen
- 反馈 ID 规则：feedback_id = motor_id + 0x10

### 通信协议

- 这台臂用的是达妙私有 CAN 协议（MIT 兼容），不是工业标准 CANopen
- 对于 7 个关节的桌面机械臂，CAN 总线完全够用
- 人形机器人（20+关节）才需要 EtherCAT，那是另一个层级的东西
- 达妙电机支持 CAN FD（波特率 > 1Mbps 自动切换），但默认 CAN 2.0B

### 供电

- 24V / 14.6A / 350W 电源，日常够用
- 7 个电机理论峰值总功耗 1296W，但不可能同时满载
- 正常工作 100-200W，高负载 200-300W

### 性能瓶颈

- 2 号电机（肩部 DM4340P）是瓶颈，长时间高负载容易过热
- 推荐：负载 ≤ 1.5kg，臂展 < 70%，连续工作 2 小时休息 15 分钟
- 建议加主动散热

### 环境踩坑

- `pyproject.toml` 里 `requires-python = "==3.10.*"` 太严格，改成 `>=3.10` 就能用 3.12
- **pip install pin 在 Windows 上会卡死**，因为要从源码编译 C++ 库
- 解决方案：用 WSL2（Ubuntu）或 conda 安装，有编译好的二进制包
- WSL2 默认装 C 盘，但可以通过 import/export 方式装到 F 盘

### WSL2 安装踩坑

- `pip install pin` 在 Windows 上从源码编译 C++，会卡死不动，根本装不上
- 解决方案：用 WSL2 Ubuntu，Linux 下 pip 有预编译的二进制包

- 微软 appx 包里套了两层：外层 appx 里有 ARM64 和 x64 两个子 appx，要解压 x64 那个才能找到 install.tar.gz
- 进入 WSL 后提示代理警告（localhost proxy not mirrored），不影响使用

### MeshCat 可视化三件套（无硬件学习的最佳工具）

#### 环境问题：WSL2 访问 Windows 文件的 Git symlink 失效

- **现象**：`python3 example/sim/fk_sim.py` 报错 `Mesh package://reBot-DevArm_description_fixend/meshes/base_link.STL could not be found`
- **原因**：仓库里 `urdf/reBot-DevArm_description_fixend` 是指向 `reBot-DevArm_fixend_description` 的软链接（两个目录名只是单词顺序不同）
  - Linux/Mac 下 git clone 会还原成真·软链接 → Pinocchio 正常工作
  - **Windows 下 Git 把软链接存成了纯文本文件**（31 字节，内容就是目标路径字符串）
  - WSL2 通过 `/mnt/f` 访问 Windows NTFS 文件系统时，看到的是那个文本文件，不是目录
- **解决方案**：在 WSL2 里手动重建软链接
  ```bash
  cd urdf
  rm reBot-DevArm_description_fixend
  ln -s reBot-DevArm_fixend_description reBot-DevArm_description_fixend
  ```
- **验证**：`ls -la` 看到 `lrwxrwxrwx ... reBot-DevArm_description_fixend -> reBot-DevArm_fixend_description` 就对了
- **教训**：WSL2 + Windows Git 混用时，软链接是个坑；遇到"文件找不到"先检查是否是 symlink 问题

- **功能**：输入 6 个关节角（度）→ 浏览器里实时显示机械臂 3D 姿态
- **启动**：`python3 example/sim/fk_sim.py` → 终端打印 `http://127.0.0.1:7000/static/` → Windows 浏览器打开

  - 输入 `0 -90 0 0 0 0` → J2 弯 90 度（大臂水平）

#### 2. ik_sim.py — 逆运动学交互式可视化（已改进）⭐

- **功能**：输入目标位置 (x, y, z) 或位姿 (x, y, z, roll, pitch, yaw) → 机械臂自动求解关节角并摆到目标位置
- **启动**：`python3 example/sim/ik_sim.py`
- **原版问题**：
  - 代码里只调用了 `viz.update(result.q)`，**没调用 `viz.show_ik_pose()` 显示目标点标记**（红球 + 三色坐标轴）
  - 每次从零位 `[0,0,0,0,0,0]` 开始求解（`q_init=None`），不使用上一次的解作为初值
  - 这导致很多点容易陷入局部最优 → 报 `[未收敛]`
  - 例如 `(0.25, 0.0, 0.15)` 误差 2.15cm，`(0.2, 0.0, 0.25)` 误差 9.46cm
- **改进方案**（已实施）：
  1. **添加目标点可视化**：调用 `viz.show_ik_pose(target_pos, target_rot, result.q)` 显示红球 + 坐标轴
  2. **连续初值策略**：保存 `q_current`，每次从上一个解出发求解下一个目标
  3. **调试输出增强**：解析失败时显示原始输入和分割结果

**URDF 里 reBot-DevArm 的关节限位**（`urdf/reBot-DevArm_fixend.urdf`）：

| 关节 | URDF 字段 | 实际范围 | 备注 |

仿真把机械臂掰成了真机不可能的姿态，质心被甩到肩关节后方，杠杆变长，力矩自然爆表。

**Pinocchio 的默认行为**：
- `pin.forwardKinematics()` / `pin.computeGeneralizedGravity()` **不会检查限位**，给什么 q 算什么
- 限位信息存在 `model.lowerPositionLimit` / `model.upperPositionLimit`（np.ndarray，弧度）
- 用户代码自己负责 validate

1. 启动时打印所有关节的 URDF 限位
2. 每次输入后用 `check_joint_limits()` 检查，超限**只警告不阻止**（仿真允许"看一下超限会怎样"是有学习价值的）
3. 推荐姿态全部改成满足限位的版本：
   ```
   0 0 0 0 0 0
   0 -90 0 0 0 0
   0 -45 -90 0 0 0      # 原来错写成 +90，肘要往后收
   0 -60 -60 30 30 0    # 原来错写成 +60
   0 -120 -30 0 0 0
   ```

**更深层收获**：
1. **运动学/动力学计算都不带限位检查**——这是工业惯例（计算和约束分离）
2. **真机 vs 仿真的鸿沟**：仿真能算的，真机未必做得到（限位、碰撞、奇异点）

## 2026-04-23：10 号算法——末端速度锁止版重力补偿

### 这个算法是什么？

`10_gravity_compensation_lock.py` 是一个**拖动示教控制器**，让机械臂在重力补偿的基础上实现"轻推能动、停手就锁"的效果。

**核心思路**：
- 持续计算末端执行器的**线速度 v_ee** 和**角速度 w_ee**
- 当末端速度 **低于阈值** 时：目标关节角度 `q_target` 保持锁定（被推也纹丝不动）
- 当末端速度 **超过阈值** 时：目标关节角度 `q_target` 更新为当前关节角度 `q`（用力推才能换位置）

**控制律**（MIT 模式）：
```
τ = τ_g(q) + kp·(q_target - q) + kd·(0 - q̇) + I
```

其中：
- `τ_g(q)` = 重力补偿力矩（Pinocchio 计算）
- `kp·(q_target - q)` = 位置误差反馈（kp=8.0）
- `kd·(0 - q̇)` = 速度阻尼（kd=1.5，目标速度为 0）
- `I` = 积分项（误差累积修正，限幅 ±0.5 N·m）

---

### 解决什么问题？

**问题 2：纯重力补偿（09 号）的缺陷**
- 09 号算法只抵消重力，没有位置锁定，机械臂会"飘"（微小扰动就漂移）
- 用户松手后，机械臂不会保持在当前位置

**10 号的解决方案**：
- 用**末端速度**作为"用户正在推"的判据（而不是关节力矩或位置误差）
- 速度超阈值 → 认为用户在推 → 更新 `q_target` 跟随当前位置
- 速度低于阈值 → 认为用户松手 → 锁定 `q_target` 不变

这样就实现了"轻推能动、停手就锁"。

---

### 怎么用？

**硬件要求**：
- 机械臂已连接并使能
- 电机支持 MIT 模式（达妙 DM-J4310-2EC 支持）

**运行**：
```bash
cd software/reBotArm_control_py
python example/10_gravity_compensation_lock.py
```

**预期行为**：
1. 启动后，机械臂锁定在当前位置（`q_target` = 初始关节角度）
2. 用力推末端执行器 → 末端速度超过阈值（0.04 m/s 或 0.08 rad/s）→ `q_target` 更新为当前角度 → 机械臂跟随你的推动
3. 松手 → 末端速度降到阈值以下 → `q_target` 锁定 → 机械臂保持在当前位置
4. Ctrl+C 停止

### 关键代码解析

#### 1. 末端速度计算（L105-110）

```python
pin.computeJointJacobians(_model, _data, q)
pin.updateFramePlacements(_model, _data)
J = pin.getFrameJacobian(_model, _data, _ee_frame_id, pin.ReferenceFrame.WORLD)
v_spatial = J @ qd
v_ee_norm = float(np.linalg.norm(v_spatial[:3]))
w_ee_norm = float(np.linalg.norm(v_spatial[3:]))
```

- `J` 是末端执行器的**雅可比矩阵**（6×6），把关节速度 `q̇` 映射到末端空间速度
- `v_spatial = J @ qd` 得到 6D 空间速度：前 3 维是线速度 `v_ee`，后 3 维是角速度 `w_ee`
- 计算速度的**范数**（模长）作为判据

#### 2. 锁止逻辑（L112-117）

```python
if v_ee_norm > _VEL_THRESHOLD or w_ee_norm > _W_VEL_THRESHOLD:
    _q_target = q.copy()
    _lock_counter = 0
    _integral *= 0.9  # 速度超阈值时积分衰减，避免锁止后误差积累
else:
    _lock_counter += 1
```

- 速度超阈值 → 更新 `q_target` 为当前角度 `q`（跟随）
- 速度低于阈值 → 保持 `q_target` 不变（锁定）
- 积分项在速度超阈值时衰减 0.9 倍，避免锁止后误差积累导致抖动

#### 3. 积分项修正（L98-103）

```python
if _integral is None:
    _integral = np.zeros_like(q)

_integral += q_error * 1.0   # 积分增益 1.0，每个控制周期累积
np.clip(_integral, -0.5, 0.5, out=_integral)  # 限幅
```

- 积分项用于修正重力补偿的**静态误差**（比如 URDF 质量参数不准、摩擦力）
- 每个控制周期（2ms）累积一次：`_integral += q_error * 1.0`
- 限幅 ±0.5 N·m，避免积分饱和

#### 4. MIT 模式控制（L119-125）

```python
arm.mit(
    pos=_q_target,
    vel=np.zeros(arm.num_joints),
    kp=np.full(arm.num_joints, _KP),
    kd=np.full(arm.num_joints, _KD),
    tau=tau_g + _integral,
)
```

- `pos=_q_target`：目标关节角度（锁止或跟随）
- `vel=0`：目标关节速度为 0（阻尼）
- `kp=8.0`：位置增益（太大推不动，太小锁不住）
- `kd=1.5`：速度阻尼（抑制振荡）
- `tau=tau_g + _integral`：前馈力矩 = 重力补偿 + 积分修正

### 可调参数

| 参数 | 默认值 | 作用 | 调参建议 |
|------|--------|------|----------|
| `_VEL_THRESHOLD` | 0.04 m/s | 末端线速度阈值 | 太大 → 推很用力才响应；太小 → 轻微抖动就解锁 |
| `_W_VEL_THRESHOLD` | 0.08 rad/s | 末端角速度阈值 | 同上 |
| `_KP` | 8.0 | 位置增益 | 太大 → 推不动；太小 → 锁不住 |
| `_KD` | 1.5 | 速度阻尼 | 太大 → 阻尼感强；太小 → 振荡 |
| 积分增益 | 1.0 | 误差累积速度 | 太大 → 积分饱和快；太小 → 静态误差大 |
| 积分限幅 | ±0.5 N·m | 积分项上限 | 太大 → 积分饱和后抖动；太小 → 修正能力弱 |

### 学习收获

1. **末端速度作为判据**：比关节力矩或位置误差更直观，避免了力传感器的需求
2. **积分项的作用**：修正重力补偿的静态误差（URDF 参数不准、摩擦力），但需要限幅避免饱和
3. **雅可比矩阵的应用**：`v_spatial = J @ qd` 把关节空间速度映射到末端空间速度，是机器人学的核心工具
4. **MIT 模式的灵活性**：可以同时指定位置、速度、增益、前馈力矩，适合复杂控制律
